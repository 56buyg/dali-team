## 沟通与响应流转机制

- 所有 agent 回复时必须遵守「流转与接棒铁律」：未完成任务不得出现无 `@` 的断流回复
- 标准传棒链路：需求输入 → 唯一理智Leader(拆解) → 视觉设计师(出图) → 最强全栈/大师前端(写代码) → 完美测试(QA) → 全能文秘(归档)
- 驳回时原路 `@` 退回上一步，附带明确的修改要求
- 阻塞时立即 `@唯一理智Leader` 请求决策，不得空等

## 代码与设计全局规范

### 分支与合入铁律

- **开工前必须拉最新 main**：`git pull origin main`，确保本地基于最新代码
- **提交前必须 rebase**：`git rebase origin/main`，禁止未经 rebase 的分支直接提 PR
- **禁止基于旧分支开发**：如果本地 main 落后于 origin/main，先同步再开工
- **合入时使用 cherry-pick**：如果分支基版本过旧，只 cherry-pick 实际变更的 commit，避免冲掉他人的代码

### 飞书操作规范（Bot 身份统一）

- **全队飞书操作统一使用 bot 身份**：`--as bot`（App ID: `cli_aaa13b30bef81ceb`）
- **放弃 device OAuth CLI 授权流程**：不再使用 `lark-cli` 的 OAuth device flow 弹授权链接
- **适用范围**：所有飞书知识库、文档读写、自动化操作均通过 bot 应用身份执行
- **来源**：大力夯（谭晓丽）提供 bot 应用凭证，统一团队操作入口（DAL-69，2026-06-22）

### 视觉与功能分离原则

- 视觉变更（CSS/组件样式/布局）与功能变更（API/逻辑/路由）应在独立分支中进行
- 目视"回归"应立即上报 @唯一理智Leader
