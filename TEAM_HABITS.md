## 沟通与响应流转机制

- 所有 agent 回复时必须遵守「流转与接棒铁律」：未完成任务不得出现无 `@` 的断流回复
- 标准传棒链路：需求输入 → 唯一理智Leader(拆解) → 视觉设计师(出图) → 最强全栈/大师前端(写代码) → 完美测试(QA) → 全能文秘(归档)
- 驳回时原路 `@` 退回上一步，附带明确的修改要求
- 阻塞时立即 `@唯一理智Leader` 请求决策，不得空等

## 代码与设计全局规范

### 开发交付流程

1. **Push + 自测后再交 QA**：修复/功能完成后，必须先 `git push` 到远程分支，并在 push 后的代码上自测核心路径，确认通过后方可 @完美测试 验收。严禁未 push 或未自测就直接交给 QA。
2. **交 QA 时注明分支和 commit**：@完美测试 时必须附带远程分支名和最新 commit hash，方便 QA 精确定位验收版本。

## 历史重大避坑

- **TEAM_HABITS.md 可读性问题**：各 agent 运行在独立 worktree 中，不能依赖"根目录"路径读取共享文件。统一通过 `multica repo checkout` 获取 `dali-team` 仓库，从仓库根目录读取 TEAM_HABITS.md 和 TASK_BOARD.md。
- **Push 再叫 QA 硬规则（DAL-52 教训）**：开发者完成修复后，必须 **push 代码到远程分支 + 自测 push 后的效果**，然后才能 @完美测试 验收。不 push 不自测就交给 QA，QA 验收的是旧代码，等于白验收，浪费时间。此规则写入代码与设计全局规范。
