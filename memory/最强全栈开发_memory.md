## 避坑指南

### Runninghub API modelId 必须精确匹配
- **坑**：文生图使用了错误的 modelId `2048642665947860994`，任务提交成功但永远不完成，导致前端轮询超时。正确 ID 是 `2048647046302801921`。
- **教训**：排查"超时"类 Bug 时，不能只看超时阈值是否合理，还要验证上游 API 的参数是否正确。模型 ID 错一个字符，API 不会报错但任务永远卡在 pending。
- **经验**：所有 Runninghub modelId 都应以平台给出的确认为准，不能随意填写或猜测。

### Bug 排查链路
- 对于异步任务类接口（提交 → 轮询 → 完成），排查顺序应为：
  1. 确认提交参数（modelId、inputs）是否正确
  2. 确认提交后任务状态是否正常推进
  3. 最后再检查超时阈值是否合理

### Runninghub 官方 API 端点与鉴权（2026-06-15 重大修复）
- **坑**：代码中的 API 端点 `/api/task/submit` 和鉴权方式与官方文档不一致，导致 `TOKEN_INVALID` (412) 错误。即使用户的 API Key 完全有效，也会被拒绝。
- **官方规范**：
  - 端点: `POST /task/openapi/create`（注意无 `/api` 前缀）
  - 鉴权: Header `Bearer <key>` **+** Body 中 `apiKey` 字段双重校验
  - 参数名: `workflowId`（非 `modelId`）
- **教训**：对接第三方 API 时，每次都要以最新官方文档为准，不能凭经验猜测端点路径和鉴权方式。同样一个 412 错误，可能是 Token 过期，也可能是端点/鉴权方式不对。
